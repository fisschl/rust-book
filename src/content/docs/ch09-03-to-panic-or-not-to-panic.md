那么，你如何决定何时应该调用 `panic!` 以及何时应该返回 `Result`？当代码 panic 时，没有办法恢复。你可以对任何错误情况调用 `panic!`，无论是否有恢复的可能，但你是在代表调用代码做出某个情况不可恢复的决定。当你选择返回 `Result` 值时，你给予调用代码选择权。调用代码可以选择以适合其情况的方式尝试恢复，或者它可以决定在这种情况下 `Err` 值是不可恢复的，因此它可以调用 `panic!` 并将你的可恢复错误转变为不可恢复的错误。因此，当你定义一个可能失败的函数时，返回 `Result` 是一个很好的默认选择。

在示例、原型代码和测试等情况下，编写会 panic 的代码而不是返回 `Result` 更合适。让我们探讨为什么，然后讨论编译器无法告诉你失败是不可能的但你作为人类可以知道的情况。本章将以一些关于如何在库代码中决定是否 panic 的一般指导方针作为结论。

## 示例、原型代码和测试

当你编写示例来说明某个概念时，同时包含健壮的错误处理代码可能会使示例不够清晰。在示例中，人们理解对像 `unwrap` 这样可能 panic 的方法的调用是作为一种占位符，表示你希望你的应用程序处理错误的方式，这可以根据你代码的其他部分正在做什么而有所不同。

类似地，当你正在原型设计且尚未决定如何处理错误时，`unwrap` 和 `expect` 方法非常方便。它们在你的代码中留下了清晰的标记，供你准备使程序更健壮时使用。

如果方法调用在测试中失败，你希望整个测试都失败，即使该方法不是正在测试的功能。因为 `panic!` 是测试被标记为失败的方式，调用 `unwrap` 或 `expect` 正是应该发生的事情。

## 当你比编译器拥有更多信息时

当你有其他逻辑确保 `Result` 将具有 `Ok` 值，但该逻辑不是编译器能理解的东西时，调用 `expect` 也是合适的。你仍然会有一个需要处理的 `Result` 值：无论你正在调用的操作在一般情况下仍有可能失败，即使在你的特定情况下逻辑上是不可能的。如果你可以通过手动检查代码确保你永远不会有 `Err` 变体，那么调用 `expect` 并在参数文本中记录你认为永远不会有 `Err` 变体的原因是完全可接受的。这里有一个例子：

```rust
use std::net::IpAddr;

let home: IpAddr = "127.0.0.1"
    .parse()
    .expect("硬编码的 IP 地址应该是有效的");
```

我们通过解析一个硬编码的字符串来创建一个 `IpAddr` 实例。我们可以看到 `127.0.0.1` 是一个有效的 IP 地址，所以在这里使用 `expect` 是可以接受的。然而，拥有一个硬编码的有效字符串并不会改变 `parse` 方法的返回类型：我们仍然得到一个 `Result` 值，编译器仍会让我们像处理 `Err` 变体是可能的情况一样处理 `Result`，因为编译器不够智能，看不出这个字符串总是一个有效的 IP 地址。如果 IP 地址字符串来自用户而不是硬编码到程序中，因此 _确实_ 有失败的可能性，我们肯定会希望以更健壮的方式处理 `Result`。提到这个 IP 地址是硬编码的假设将提示我们，如果在未来我们需要从其他来源获取 IP 地址，要将 `expect` 更改为更好的错误处理代码。

## 错误处理指导方针

建议当你的代码可能最终处于不良状态时，让你的代码 panic。在这种上下文中，_不良状态_是指某些假设、保证、契约或不变量被打破，例如当无效值、矛盾值或缺失值被传递给你的代码时——加上以下一项或多项：

- 不良状态是意外的，而不是像用户以错误格式输入数据那样可能偶尔发生的事情。
- 你之后的代码需要依赖不处于这种不良状态，而不是在每一步都检查问题。
- 没有一种好的方式在你使用的类型中编码这些信息。我们将在第 18 章的[将状态和行为编码为类型][encoding]中介绍一个示例。

如果有人调用你的代码并传入没有意义的值，最好返回一个错误，这样库的用户可以决定他们在这种情况下想做什么。然而，在继续可能不安全或有害的情况下，最好的选择可能是调用 `panic!` 并提醒使用你的库的人注意他们代码中的 bug，以便他们可以在开发期间修复它。类似地，如果你正在调用你无法控制的外部代码并且它返回一个你无法修复的无效状态，`panic!` 通常是合适的。

然而，当失败是预期的情况时，返回 `Result` 比进行 `panic!` 调用更合适。示例包括解析器被给予格式错误的数据，或 HTTP 请求返回一个状态，表明你已达到速率限制。在这些情况下，返回 `Result` 表明失败是调用代码必须决定如何处理的预期可能性。

当你的代码执行的操作如果使用无效值调用可能会使用户处于风险中时，你的代码应该首先验证值是否有效，如果值无效则 panic。这主要是出于安全原因：尝试对无效数据进行操作可能会使你的代码暴露于漏洞。这是标准库在你尝试越界内存访问时会调用 `panic!` 的主要原因：尝试访问不属于当前数据结构的内存是一个常见的安全问题。函数通常有 _契约_ ：只有当输入满足特定要求时，其行为才有保证。当契约被违反时 panic 是有意义的，因为契约违反总是表明调用方的 bug，而且这不是你希望调用代码必须显式处理的错误类型。事实上，没有合理的方式让调用代码恢复；调用_程序员_需要修复代码。函数的契约，特别是当违规将导致 panic 时，应该在函数的 API 文档中解释；我们将在第 14 章介绍在你创建的 API 文档中指示 `panic!` 可能性的文档约定。

然而，在你所有的函数中都有大量的错误检查将是冗长且烦人的。幸运的是，你可以使用 Rust 的类型系统（以及编译器完成的类型检查）为你进行许多检查。如果你的函数有一个特定类型作为参数，你可以带着编译器已经确保你拥有有效值的知识继续你代码的逻辑。例如，如果你有一个类型而不是 `Option`，你的程序期望有 _某些东西_ 而不是 _没有东西_ 。你的代码然后不必处理 `Some` 和 `None` 变体的两种情况：它将只有确定拥有一个值的一种情况。试图将 nothing 传递给你的函数的代码甚至不会编译，所以你的函数不必在运行时检查这种情况。另一个例子是使用无符号整数类型如 `u32`，它确保参数永远不会是负数。

## 用于验证的自定义类型

让我们进一步探讨使用 Rust 的类型系统确保我们拥有有效值的想法，并看看创建用于验证的自定义类型。回想一下第 2 章中的猜数字游戏，我们的代码要求用户猜测 1 到 100 之间的数字。我们从未验证用户的猜测是否在这些数字之间，然后才将其与我们的秘密数字进行检查；我们只验证了猜测是正数。在这种情况下，后果不是很严重：我们的"太大"或"太小"输出仍然是正确的。但引导用户进行有效猜测，并在用户猜测超出范围的数字与用户输入字母（例如）时有不同的行为，将是一个有用的增强。

一种方法是将猜测解析为 `i32` 而不是仅仅 `u32` 以允许潜在的负数，然后添加一个检查数字是否在范围内的检查，如下所示：

**src/main.rs**

```rust
use rand::Rng;
use std::cmp::Ordering;
use std::io;

fn main() {
    println!("猜数字！");

    let secret_number = rand::thread_rng().gen_range(1..=100);

    loop {
        // --snip--

        println!("请输入你的猜测。");

        let mut guess = String::new();

        io::stdin()
            .read_line(&mut guess)
            .expect("读取行失败");

        let guess: i32 = match guess.trim().parse() {
            Ok(num) => num,
            Err(_) => continue,
        };

        if guess < 1 || guess > 100 {
            println!("秘密数字将在 1 和 100 之间。");
            continue;
        }

        match guess.cmp(&secret_number) {
            // --snip--
            Ordering::Less => println!("太小了！"),
            Ordering::Greater => println!("太大了！"),
            Ordering::Equal => {
                println!("你赢了！");
                break;
            }
        }
    }
}
```

`if` 表达式检查我们的值是否超出范围，告诉用户问题，并调用 `continue` 开始循环的下一次迭代并要求另一个猜测。在 `if` 表达式之后，我们可以继续进行 `guess` 和秘密数字之间的比较，知道 `guess` 在 1 和 100 之间。

然而，这不是一个理想的解决方案：如果程序只在 1 到 100 之间的值上操作是绝对关键的，并且它有许多有这个要求的函数，在每个函数中都有这样的检查将是乏味的（而且可能影响性能）。

相反，我们可以在专用模块中创建一个新类型，并在函数中放置验证以创建该类型的实例，而不是到处重复验证。这样，函数在其签名中使用新类型并自信地使用它们接收到的值是安全的。代码示例 9-13 展示了定义 `Guess` 类型的一种方式，只有当 `new` 函数接收到 1 到 100 之间的值时才会创建 `Guess` 的实例。

**代码示例 9-13：只有当值在 1 和 100 之间时才会继续的 `Guess` 类型**

```rust
pub struct Guess {
    value: i32,
}

impl Guess {
    pub fn new(value: i32) -> Guess {
        if value < 1 || value > 100 {
            panic!("猜测值必须在 1 和 100 之间，得到的是 {value}。");
        }

        Guess { value }
    }

    pub fn value(&self) -> i32 {
        self.value
    }
}
```

请注意，*src/guessing_game.rs* 中的这段代码依赖于在 *src/lib.rs* 中添加模块声明 `mod guessing_game;`，我们在这里没有展示。在这个新模块的文件中，我们定义了一个名为 `Guess` 的结构体，它有一个名为 `value` 的字段，保存一个 `i32`。这是数字将被存储的地方。

然后，我们在 `Guess` 上实现一个名为 `new` 的关联函数，用于创建 `Guess` 值的实例。`new` 函数被定义为有一个名为 `value`、类型为 `i32` 的参数，并返回一个 `Guess`。`new` 函数体中的代码测试 `value` 以确保它在 1 和 100 之间。如果 `value` 没有通过此测试，我们进行 `panic!` 调用，这将提醒编写调用代码的程序员他们有 bug 需要修复，因为创建一个 `value` 超出此范围的 `Guess` 将违反 `Guess::new` 所依赖的契约。`Guess::new` 可能 panic 的情况应该在其面向公众的 API 文档中讨论；我们将在第 14 章介绍在你创建的 API 文档中指示 `panic!` 可能性的文档约定。如果 `value` 确实通过了测试，我们创建一个 `value` 字段设置为 `value` 参数的新 `Guess` 并返回 `Guess`。

接下来，我们实现一个名为 `value` 的方法，该方法借用 `self`，没有其他参数，并返回一个 `i32`。这种方法有时被称为 _获取器_ ，因为它的目的是从其字段获取一些数据并返回它。这个公共方法是必要的，因为 `Guess` 结构体的 `value` 字段是私有的。`value` 字段必须是私有的，以便使用 `Guess` 结构体的代码不被允许直接设置 `value`：`guessing_game` 模块之外的代码 _必须_ 使用 `Guess::new` 函数来创建 `Guess` 的实例，从而确保没有 `Guess` 可以拥有未经 `Guess::new` 函数中条件检查的 `value`。

一个参数或只返回 1 到 100 之间数字的函数然后可以在其签名中声明它接受或返回 `Guess` 而不是 `i32`，并且不需要在其主体中进行任何额外的检查。

## 总结

Rust 的错误处理功能旨在帮助你编写更健壮的代码。`panic!` 宏表明你的程序处于它无法处理的状态，并让你告诉进程停止而不是尝试继续使用无效或不正确的值。`Result` 枚举使用 Rust 的类型系统来指示操作可能以你的代码可以恢复的方式失败。你可以使用 `Result` 来告诉调用你的代码的代码，它也需要处理潜在的成功或失败。在适当的情况下使用 `panic!` 和 `Result` 将使你的代码在面对不可避免的问题时更可靠。
