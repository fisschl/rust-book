title: "Rc<T>，引用计数的智能指针"

在大多数情况下，所有权是明确的：你确切知道哪个变量拥有给定的值。然而，有些情况下单个值可能有多个所有者。例如，在图数据结构中，多个边可能指向同一个节点，该节点在概念上由所有指向它的边拥有。除非没有任何边指向它，因此没有所有者，否则不应清理节点。

你必须通过使用 Rust 类型 `Rc<T>` 显式启用多重所有权，它是 _reference counting_ 的缩写。`Rc<T>` 类型跟踪对值的引用数量，以确定该值是否仍在使用。如果对值的引用数量为零，则可以在没有任何引用变为无效的情况下清理该值。

将 `Rc<T>` 想象成家庭房间里的电视。当一个人进入房间看电视时，他们会打开它。其他人可以进入房间观看电视。当最后一个人离开房间时，他们关掉电视，因为它不再被使用了。如果有人在其他人还在观看时关掉电视，其余的电视观看者会发出一片哗然！

当我们想在堆上分配一些数据供程序的多个部分读取，并且我们无法在编译时确定哪部分最后使用完数据时，我们使用 `Rc<T>` 类型。如果我们知道哪部分会最后完成，我们可以让那部分成为数据的所有者，编译时强制执行的正常所有权规则就会生效。

注意，`Rc<T>` 仅用于单线程场景。当我们讨论第16章的并发时，我们将介绍如何在多线程程序中进行引用计数。

### 共享数据

让我们回到清单 15-5 中的 cons list 示例。回想一下，我们使用 `Box<T>` 定义了它。这次，我们将创建两个都共享第三个列表所有权的列表。从概念上讲，这看起来类似于图 15-3。

我们将创建包含 `5` 然后 `10` 的列表 `a`。然后，我们将再创建两个列表：`b` 从 `3` 开始，`c` 从 `4` 开始。然后 `b` 和 `c` 列表都将继续指向包含 `5` 和 `10` 的第一个 `a` 列表。换句话说，两个列表将共享包含 `5` 和 `10` 的第一个列表。

尝试使用我们使用 `Box<T>` 的 `List` 定义来实现这个场景是行不通的，如清单 15-17 所示。

**代码示例 15-17**：演示我们不允许有两个使用 `Box<T>` 的列表尝试共享第三个列表的所有权

```rust
enum List {
    Cons(i32, Box<List>),
    Nil,
}

use crate::List::{Cons, Nil};

fn main() {
    let a = Cons(5, Box::new(Cons(10, Box::new(Nil))));
    let b = Cons(3, Box::new(a));
    let c = Cons(4, Box::new(a));
}
```

`Cons` 变体拥有它们持有的数据，因此当我们创建 `b` 列表时，`a` 被移入 `b` 并且 `b` 拥有 `a`。然后，当我们尝试在创建 `c` 时再次使用 `a` 时，我们不被允许，因为 `a` 已经被移动了。

我们可以更改 `Cons` 的定义以持有引用，但那样我们就必须指定生命周期参数。通过指定生命周期参数，我们将指定列表中的每个元素将至少与整个列表一样长。这在清单 15-17 中的元素和列表中是这种情况，但并非在每个场景中都如此。

相反，我们将更改 `List` 的定义以使用 `Rc<T>` 代替 `Box<T>`，如清单 15-18 所示。每个 `Cons` 变体现在将持有一个值和一个指向 `List` 的 `Rc<T>`。当我们创建 `b` 时，我们不取得 `a` 的所有权，而是克隆 `a` 持有的 `Rc<List>`，从而将引用数量从1增加到2，并让 `a` 和 `b` 共享该 `Rc<List>` 中数据的所有权。在创建 `c` 时我们也将克隆 `a`，将引用数量从2增加到3。每次我们调用 `Rc::clone` 时，`Rc<List>` 内数据的引用计数将增加，并且除非有零个引用指向它，否则数据不会被清理。

**代码示例 15-18**：使用 `Rc<T>` 的 `List` 定义

```rust
enum List {
    Cons(i32, Rc<List>),
    Nil,
}

use crate::List::{Cons, Nil};
use std::rc::Rc;

fn main() {
    let a = Rc::new(Cons(5, Rc::new(Cons(10, Rc::new(Nil)))));
    let b = Cons(3, Rc::clone(&a));
    let c = Cons(4, Rc::clone(&a));
}
```

我们需要添加 `use` 语句以将 `Rc<T>` 引入作用域，因为它不在 prelude 中。在 `main` 中，我们创建持有 `5` 和 `10` 的列表并将其存储在 `a` 中的新 `Rc<List>` 中。然后，当我们创建 `b` 和 `c` 时，我们调用 `Rc::clone` 函数并将对 `a` 中 `Rc<List>` 的引用作为参数传递。

我们可以调用 `a.clone()` 而不是 `Rc::clone(&a)`，但 Rust 的惯例是在这种情况下使用 `Rc::clone`。`Rc::clone` 的实现不像大多数类型的 `clone` 实现那样对所有数据进行深拷贝。对 `Rc::clone` 的调用只增加引用计数，这不需要太多时间。数据的深拷贝可能需要很长时间。通过使用 `Rc::clone` 进行引用计数，我们可以在视觉上区分深拷贝类型的克隆和增加引用计数类型的克隆。在寻找代码中的性能问题时，我们只需要考虑深拷贝克隆，可以忽略对 `Rc::clone` 的调用。

### 克隆以增加引用计数

让我们更改清单 15-18 中的工作示例，以便我们可以看到在创建和释放对 `a` 中 `Rc<List>` 的引用时引用计数的变化。

在清单 15-19 中，我们将更改 `main` 使其在列表 `c` 周围有一个内部作用域；然后，我们可以看到当 `c` 超出作用域时引用计数如何变化。

**代码示例 15-19**：打印引用计数

```rust
enum List {
    Cons(i32, Rc<List>),
    Nil,
}

use crate::List::{Cons, Nil};
use std::rc::Rc;

fn main() {
    let a = Rc::new(Cons(5, Rc::new(Cons(10, Rc::new(Nil)))));
    println!("count after creating a = {}", Rc::strong_count(&a));
    let b = Cons(3, Rc::clone(&a));
    println!("count after creating b = {}", Rc::strong_count(&a));
    {
        let c = Cons(4, Rc::clone(&a));
        println!("count after creating c = {}", Rc::strong_count(&a));
    }
    println!("count after c goes out of scope = {}", Rc::strong_count(&a));
}
```

在程序中引用计数变化的每个点，我们打印引用计数，我们通过调用 `Rc::strong_count` 函数获得。这个函数被命名为 `strong_count` 而不是 `count`，因为 `Rc<T>` 类型还有一个 `weak_count`；我们将在 ["使用 `Weak<T>` 防止引用循环"][preventing-ref-cycles] 中看到 `weak_count` 的用途。

此代码打印以下内容：

```console
$ cargo run
   Compiling cons-list v0.1.0 (file:///projects/cons-list)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.45s
     Running `target/debug/cons-list`
count after creating a = 1
count after creating b = 2
count after creating c = 3
count after c goes out of scope = 2
```

我们可以看到 `a` 中的 `Rc<List>` 的初始引用计数为1；然后，每次我们调用 `clone` 时，计数增加1。当 `c` 超出作用域时，计数减少1。我们不必调用函数来减少引用计数，就像我们必须调用 `Rc::clone` 来增加引用计数一样：`Drop` trait 的实现会在 `Rc<T>` 值超出作用域时自动减少引用计数。

在此示例中我们看不到的是，当 `b` 和然后在 `main` 结束时的 `a` 超出作用域时，计数为0，`Rc<List>` 被完全清理。使用 `Rc<T>` 允许单个值有多个所有者，计数确保值保持有效，只要任何所有者仍然存在。

通过不可变引用，`Rc<T>` 允许你在程序的多个部分之间共享数据，仅用于读取。如果 `Rc<T>` 也允许你有多个可变引用，你可能会违反第4章讨论的借用规则之一：对同一位置的多个可变借用可能导致数据竞争和不一致。但能够改变数据是非常有用的！在下一节中，我们将讨论内部可变性模式和 `RefCell<T>` 类型，你可以将其与 `Rc<T>` 结合使用来处理这种不可变性限制。
