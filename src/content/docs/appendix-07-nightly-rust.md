本附录是关于 Rust 是如何构建的，以及这如何影响你作为 Rust 开发者。

## 稳定而不停滞

作为一种语言，Rust 非常关心代码的稳定性。我们希望 Rust 成为你构建的坚如磐石的基础，如果事物不断变化，那将是不可能的。同时，如果我们不能试验新功能，我们可能会在发布之后才发现重要的缺陷，那时我们已经无法再更改了。

我们对这个问题的解决方案就是我们所说的"稳定而不停滞"，我们的指导原则是：你永远不应该害怕升级到稳定版 Rust 的新版本。每次升级应该是无痛的，但也应该带来新功能、更少的错误和更快的编译时间。

## 咔嚓咔嚓！发布通道和搭火车

Rust 开发遵循 _火车时刻表_。也就是说，所有开发都在 Rust 仓库的主分支上进行。发布遵循软件发布火车模型，Cisco IOS 和其他软件项目都使用过这种模型。Rust 有三个 _发布通道_：

- Nightly
- Beta
- Stable

大多数 Rust 开发者主要使用稳定通道，但那些想要尝试实验性新功能的人可能会使用 nightly 或 beta。

以下是开发和发布过程如何工作的一个示例：假设 Rust 团队正在开发 Rust 1.5 的发布。该发布发生在 2015 年 12 月，但它将为我们提供真实的版本号。新功能被添加到 Rust 中：一个新的提交落在 nightly 使用的主分支上。每晚都会产生一个新的 Rust nightly 版本。每天都是发布日，这些发布由我们的发布基础设施自动创建。因此随着时间推移，我们的发布看起来像这样，每晚一次：

```text
nightly: * - - * - - *
```

每六周，是时候准备新版本了！Rust 仓库的 `beta` 分支从 nightly 使用的主分支分出来。现在，有两个发布：

```text
nightly: * - - * - - *
                     |
beta:                *
```

大多数 Rust 用户不积极使用 beta 版本，但在他们的 CI 系统中针对 beta 进行测试，以帮助 Rust 发现可能的回归。与此同时，每晚仍然有 nightly 发布：

```text
nightly: * - - * - - * - - * - - *
                     |
beta:                *
```

假设发现了一个回归。很好，我们在回归潜入稳定版本之前有时间测试 beta 版本！修复被应用到主分支，因此 nightly 被修复，然后修复被反向移植到 `beta` 分支，并产生一个新的 beta 版本：

```text
nightly: * - - * - - * - - * - - * - - *
                     |
beta:                * - - - - - - - - *
```

在第一个 beta 创建后六周，是时候发布稳定版了！`stable` 分支从 `beta` 分支产生：

```text
nightly: * - - * - - * - - * - - * - - * - * - *
                     |
beta:                * - - - - - - - - *
                                        |
stable:                                *
```

万岁！Rust 1.5 完成了！然而，我们忘了一件事：因为六周已经过去了，我们还需要下一个 Rust 版本 1.6 的新 beta。因此在 `stable` 从 `beta` 分支后，下一个版本的 `beta` 再次从 `nightly` 分支：

```text
nightly: * - - * - - * - - * - - * - - * - * - *
                     |                         |
beta:                * - - - - - - - - *       *
                                        |
stable:                                *
```

这被称为"火车模型"，因为每六周，一个发布"离开车站"，但仍必须通过 beta 通道的旅程才能作为稳定发布到达。

Rust 每六周发布一次，就像钟表一样。如果你知道一个 Rust 发布的日期，你就知道下一个发布的日期：六周后。拥有每六周计划发布的一个好处是下一班火车即将到来。如果一个功能碰巧错过了特定发布，不必担心：短期内还有另一个发布发生！这有助于减少在发布截止日期前偷偷塞进可能不完美的功能的压力。

感谢这个过程，你总是可以查看 Rust 的下一个构建并亲自验证它很容易升级：如果 beta 发布不如预期工作，你可以向团队报告并在下一个稳定发布发生之前修复它！beta 发布中的破坏性变更相对罕见，但 `rustc` 仍然是一个软件，确实存在错误。

## 维护时间

Rust 项目支持最新的稳定版本。当新的稳定版本发布时，旧版本达到其生命周期终止（EOL）。这意味着每个版本被支持六周。

## 不稳定功能

这个发布模型还有一个需要注意的地方：不稳定功能。Rust 使用一种称为"功能标志"的技术来确定在给定发布中启用哪些功能。如果一个新功能正在积极开发中，它会落在主分支上，因此进入 nightly，但在 _功能标志_ 后面。如果你作为用户希望尝试进行中的功能，你可以，但你必须使用 Rust 的 nightly 发布，并在源代码中用适当的标志进行注释以选择加入。

如果你使用 Rust 的 beta 或稳定版本，你不能使用任何功能标志。这是使我们能够在永远宣布它们稳定之前获得新功能的实际使用的关键。那些希望选择进入最前沿的人可以做到，那些想要坚如磐石体验的人可以坚持使用稳定版，并知道他们的代码不会崩溃。稳定而不停滞。

本书只包含有关稳定功能的信息，因为进行中的功能仍在变化，而且它们肯定会在这本书编写和它们在稳定构建中启用之间有所不同。你可以在网上找到 nightly 专用功能的文档。

## Rustup 和 Rust Nightly 的作用

Rustup 可以轻松地在全局或每个项目的基础上在 Rust 的不同发布通道之间切换。默认情况下，你将安装稳定版 Rust。要安装 nightly，例如：

```console
$ rustup toolchain install nightly
```

你也可以看到你用 `rustup` 安装的所有 _工具链_（Rust 发布和相关组件）。以下是其中一位作者的 Windows 计算机上的示例：

```powershell
> rustup toolchain list
stable-x86_64-pc-windows-msvc (default)
beta-x86_64-pc-windows-msvc
nightly-x86_64-pc-windows-msvc
```

如你所见，稳定工具链是默认的。大多数 Rust 用户大部分时间使用稳定版。你可能想大部分时间使用稳定版，但在特定项目上使用 nightly，因为你关心一个前沿功能。为此，你可以在该项目的目录中使用 `rustup override` 将 nightly 工具链设置为 `rustup` 在你处于该目录时应使用的工具链：

```console
$ cd ~/projects/needs-nightly
$ rustup override set nightly
```

现在，每次你在 _~/projects/needs-nightly_ 内调用 `rustc` 或 `cargo` 时，`rustup` 将确保你使用的是 nightly Rust，而不是你的默认稳定版 Rust。当你有很多 Rust 项目时，这非常方便！

## RFC 流程和团队

那么你如何了解这些新功能呢？Rust 的开发模型遵循 _征求意见（RFC）流程_。如果你希望 Rust 有所改进，你可以撰写一份提案，称为 RFC。

任何人都可以撰写 RFC 来改进 Rust，提案由 Rust 团队审查和讨论，该团队由许多主题子团队组成。团队完整列表[在 Rust 的网站上](https://www.rust-lang.org/governance)，包括项目每个领域的团队：语言设计、编译器实现、基础设施、文档等。适当的团队阅读提案和评论，写下自己的一些评论，最终，达成共识接受或拒绝该功能。

如果功能被接受，会在 Rust 仓库上打开一个问题，然后有人可以实现它。实现它的人很可能不是最初提出该功能的人！当实现准备好时，它会像我们在["不稳定功能"](#不稳定功能)一节中讨论的那样，落在功能标志后面的主分支上。

一段时间后，一旦使用 nightly 版本的 Rust 开发者能够试用新功能，团队成员将讨论该功能、它在 nightly 上的表现如何，并决定是否将其纳入稳定版 Rust。如果决定是向前推进，功能标志将被移除，该功能现在被视为稳定！它搭乘火车进入 Rust 的新稳定发布。
