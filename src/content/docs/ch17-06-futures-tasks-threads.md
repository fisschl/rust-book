title: 综合应用：Futures、Tasks 和 Threads
---

正如我们在[第16章][ch16]看到的，线程提供了一种并发方法。我们在本章中看到了另一种方法：使用 async 与 futures 和 streams。如果你想知道何时选择一种方法而不是另一种，答案是：这取决于！而且在很多情况下，选择不是线程 *或* async，而是线程 *和* async。

许多操作系统几十年来一直提供基于线程的并发模型，许多编程语言也因此支持它们。然而，这些模型并非没有权衡。在许多操作系统上，它们为每个线程使用相当多的内存。线程也只是当你的操作系统和硬件支持它们时的一个选项。与主流桌面和移动计算机不同，一些嵌入式系统根本没有操作系统，因此它们也没有线程。

async 模型提供了一组不同的——最终是互补的——权衡。在 async 模型中，并发操作不需要自己的线程。相反，它们可以在 tasks 上运行，就像我们在 streams 部分使用 `trpl::spawn_task` 从同步函数启动工作一样。task 类似于线程，但它不是由操作系统管理，而是由库级代码管理：运行时。

生成线程和生成 tasks 的 API 如此相似是有原因的。线程充当一组同步操作的边界；并发在*线程之间*是可能的。Tasks 充当一组*异步*操作的边界；并发在*之间*和*内部*都是可能的，因为 task 可以在其主体中的 futures 之间切换。最后，futures 是 Rust 最细粒度的并发单元，每个 future 可能代表其他 futures 的树。运行时——特别是它的执行器——管理 tasks，而 tasks 管理 futures。在这方面，tasks 类似于轻量级的、运行时管理的线程，具有因为由运行时而不是操作系统管理而带来的额外能力。

这并不意味着 async tasks 总是比线程更好（或反之亦然）。在某些方面，使用线程进行并发比使用 `async` 更简单的编程模型。这可以是一个优势或劣势。线程有些"即发即弃"；它们没有与 future 相当的原生功能，因此它们只是运行到完成，除非被操作系统本身中断。

而且事实证明，线程和 tasks 通常能很好地协同工作，因为 tasks 可以（至少在某些运行时中）在线程之间移动。事实上，在底层，我们一直在使用的运行时——包括 `spawn_blocking` 和 `spawn_task` 函数——默认是多线程的！许多运行时使用一种称为 *工作窃取* 的方法，根据线程当前的使用情况，透明地在线程之间移动 tasks，以提高系统的整体性能。那种方法实际上需要线程*和*tasks，因此也需要 futures。

在考虑何时使用哪种方法时，请考虑以下经验法则：

- 如果工作是 *高度可并行化* 的（即 CPU 密集型），比如处理一堆可以分别处理的数据，线程是更好的选择。
- 如果工作是 *高度并发* 的（即 I/O 密集型），比如处理来自许多不同来源的消息，这些消息可能以不同的间隔或不同的速率到达，async 是更好的选择。

如果你既需要并行性又需要并发性，你不必在线程和 async 之间选择。你可以自由地一起使用它们，让每个发挥它最擅长的作用。例如，清单 17-25 展示了现实世界中 Rust 代码中这种混合的一个相当常见的例子。

**清单 17-25**：在线程中使用阻塞代码发送消息，在 async 代码块中等待消息

```rust
use trpl;
use std::thread;
use std::time::Duration;

fn main() {
    let (tx, mut rx) = trpl::channel();

    thread::spawn(move || {
        for i in 1..=10 {
            tx.send(i).unwrap();
            thread::sleep(Duration::from_secs(1));
        }
    });

    trpl::block_on(async {
        while let Some(value) = rx.recv().await {
            println!("received: {}", value);
        }
    });
}
```

我们首先创建一个异步通道，然后生成一个使用 `move` 关键字获取通道发送端所有权的线程。在线程中，我们发送数字 1 到 10，每条消息之间休眠一秒。最后，我们运行一个用传递给 `trpl::block_on` 的 async 代码块创建的 future，就像我们在本章中一直做的那样。在那个 future 中，我们等待那些消息，就像我们在其他消息传递示例中看到的那样。

回到我们在本章开头提到的场景，想象使用专用线程运行一组视频编码任务（因为视频编码是 CPU 密集型的），但使用 async 通道通知 UI 这些操作已完成。现实世界的用例中有无数这样的组合示例。
