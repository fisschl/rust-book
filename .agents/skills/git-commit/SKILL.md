# Git 智能提交流程

## 概述

自动化 Git 提交流程：检查更改 → 分析可分解性 → 编写标准提交信息 → 提交。不推送（除非用户明确要求）。

## 流程

### 1. 检查当前更改

运行以下命令获取完整更改信息：

```bash
git status
git diff --staged
git diff
```

- 如果暂存区有更改（`git diff --staged` 非空），只处理暂存区的内容
- 如果暂存区为空但工作区有更改（`git diff` 非空），先执行 `git add -A` 暂存所有更改

### 2. 分析可分解性

分析更改内容，判断是否可分解为多个逻辑独立的提交：

**可分解的条件（需同时满足）：**
- 更改涉及多个不相关的文件或模块
- 每个子集有独立的逻辑意义（如：修复 bug + 新增功能 + 重构）
- 子集之间没有交叉依赖

**不可分解（大多数情况）：**
- 所有更改服务于同一个目的
- 文件之间有紧密的逻辑关联
- 更改量较小且内聚

若可分解，为每个子集分别执行步骤 3-4，分批 `git add` 后提交。

### 3. 编写提交信息

分析 `git diff --staged` 的内容，编写符合标准格式的提交信息，直接写入 `.git/COMMIT_EDITMSG`。

**格式要求：**

```
<type>(<scope>): <简短概述>

<详细说明>
```

- **概述行**：不超过 50 个中文字符（或 72 个英文字符），以句号结尾
- **空一行**
- **详细说明**：解释做了什么、为什么这样做、有什么影响。每行不超过 72 个英文字符

**type 类型：**
| type | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复 bug |
| docs | 文档更改 |
| style | 格式调整（不影响代码逻辑） |
| refactor | 重构（既非新功能也非修复） |
| perf | 性能优化 |
| test | 测试相关 |
| chore | 构建/工具/依赖更新 |
| ci | CI 配置更改 |

**编写原则：**
- 概述简洁明了，一眼能看出做了什么
- 详细说明解释变更动机、实现方式、影响范围
- 如果变更很简单，详细说明可以只有一行
- 使用中文编写

**写入方式：**
使用 `create_file` 工具将提交信息写入 `.git/COMMIT_EDITMSG`。如果文件已存在，先覆盖它。

### 4. 执行提交

```bash
git commit -F .git/COMMIT_EDITMSG
```

### 5. 确认结果

运行 `git log -1 --oneline` 确认提交成功，向用户展示提交摘要。

## 推送

**默认不推送。** 除非用户在请求中明确要求推送（如"提交并推送"、"commit and push"），否则跳过推送步骤。如果用户要求推送，执行：

```bash
git push
```

## 注意事项

- 永远不要在没有检查更改内容的情况下直接提交
- 如果 `git diff --staged` 和 `git diff` 都为空，告知用户没有可提交的更改
- 如果可分解为多个提交，逐批处理，每批单独编写提交信息
- 提交前向用户展示提交信息概览，获得确认后再执行
- 不要使用 `git commit -m`，始终通过 `.git/COMMIT_EDITMSG` 文件提交
